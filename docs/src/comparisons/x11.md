# BYO/OS for X11 Developers

X11 relies on server-managed resources, where windows reside on the display
server rather than the client. BYO/OS retains this centralized model but
reallocates the responsibility for rendering. While X11 clients paint their own
pixels and manage internal widgets, BYO/OS moves these duties to the system,
rendering the entire UI hierarchy centrally. This page maps X11 concepts to
BYO/OS.

## Architecture

X11 uses a client-server model. The X server owns the display and
handles rendering and input. A separate **window manager** — itself an
X client with special privileges (`SubstructureRedirect`) — handles
window placement, decorations, stacking, and focus policy. Other clients
connect over a socket and issue requests (create window, draw, etc.).

BYO/OS uses a hub-and-spoke model. The orchestrator is the hub — it
maintains the object tree and routes messages. The compositor is a spoke
that subscribes to object types and renders them. Apps are spokes that
create objects.

| X11 | BYO/OS |
|---|---|
| X server (display + input + resources) | Orchestrator (tree + routing) + Compositor (display) |
| Window manager (placement, focus) | Orchestrator conventions + daemons |
| Client ↔ server socket | Process ↔ orchestrator stdin/stdout |
| Binary protocol with opcodes | Text protocol with operations (`+`, `-`, `@`) |

In X11, the server handles rendering and resource tracking while the
window manager handles layout and focus as a privileged client. BYO/OS
splits differently: the orchestrator manages the tree and routes
messages, the compositor renders. Both the compositor and window
management logic are peers of the orchestrator, replaceable
independently.

## Centralized Rendering

X11 and BYO/OS both centralize rendering — apps describe what they want,
and a system process draws it.

In X11, apps issue drawing commands (XDrawLine, XFillRectangle) or
submit pixmaps. The X server executes them against its framebuffer.
Modern X11 apps mostly bypass this by rendering client-side and
submitting buffers via MIT-SHM or DRI, making X11 function more like
Wayland in practice.

In BYO/OS, apps create objects in a tree (`+view sidebar class="w-64"`).
The compositor interprets the tree and renders it via GPU. There are no
drawing primitives — the tree is the only interface.

X11's drawing model is imperative (sequences of draw commands). BYO/OS's
model is declarative (a persistent tree of objects with properties).

## Resources → Objects

X11 has server-side resources: windows, pixmaps, graphics contexts,
colormaps, fonts. Each has a numeric ID (XID). The server allocates and
tracks them; clients refer to them by ID.

BYO/OS has objects in a tree. Each has a string ID qualified by client
name (`app:sidebar`). The orchestrator tracks them.

| X11 | BYO/OS |
|---|---|
| Window (XID) | Object (`+view sidebar`) |
| Window hierarchy (parent/child) | Object tree (parent/child) |
| Window properties (`XChangeProperty`) | Object props (`class="w-64"`) |
| `XCreateWindow` | `+kind id props` (upsert) |
| `XDestroyWindow` (cascades) | `-kind id` (cascades) |
| `XChangeProperty` | `@kind id key=value` (patch) |

Both maintain a tree of resources with parent-child relationships and
key-value metadata. X11 properties are typed atoms; BYO/OS properties
are untyped strings.

## Atoms → Strings

X11 uses **atoms** — interned strings mapped to integer IDs for
efficiency (`_NET_WM_STATE`, `WM_NAME`). Clients intern once and use the
integer thereafter.

BYO/OS uses plain strings for object kinds and property names. No
interning. Simpler, but every message carries full string names.

## Events

X11 events are raw input: `KeyPress`, `ButtonPress`, `MotionNotify`,
`Expose`. The X server sends them to the focused or pointer-containing
window. Clients do their own hit testing within their window.

BYO/OS events are semantic: `!click 0 save`,
`!keydown 0 sidebar key="a"`. They target objects by ID. The compositor
does all hit testing. Apps acknowledge with a handling disposition
(`!ack click 0 handled=true`).

## Extensions → Object Types

X11 adds capabilities via protocol extensions: XRender, XComposite,
XInput2, GLX, DRI3. Each defines new opcodes, negotiated at connection
time.

BYO/OS adds capabilities by creating new object types. No negotiation,
no opcode allocation. A daemon claims a type string and handles it. The
trade-off: no formal contract for what a type supports yet.

## Network Transparency

Both are network-transparent.

X11 runs over TCP or Unix socket. Remote X11 works but is
bandwidth-heavy for modern apps. SSH X forwarding is common but slow for
complex UIs.

BYO/OS runs over stdin/stdout. SSH provides remote display by piping
streams. The text protocol is compact for UI descriptions. Raster
content (Kitty graphics) needs the same bandwidth as X11 pixmaps.

## ICCCM / EWMH → Orchestrator Conventions

X11 relies on **conventions** for inter-client communication: ICCCM
(window management hints, selections) and EWMH (desktop, taskbar,
fullscreen). These are agreements between clients and window managers,
not enforced by the X server.

BYO/OS handles some of these at the orchestrator level (daemon claims,
observer subscriptions, expansion) and will define conventions for
others (window management, clipboard). The orchestrator can enforce
conventions since it controls the tree.

## Concept Map

| X11 | BYO/OS |
|---|---|
| X server | Orchestrator + compositor |
| Window manager | Orchestrator conventions + daemons |
| Window (XID) | Object (qualified ID) |
| Window hierarchy | Object tree |
| `XChangeProperty` / `XGetProperty` | Props (`key=value`, `@` patch) |
| Atom | Kind string / prop name (no interning) |
| Drawing commands | No equivalent (declarative tree) |
| Extension (XRender, etc.) | New object type (string) |
| `KeyPress` / `ButtonPress` | `!keydown` / `!click` |
| Selection (clipboard) | Not yet defined |
| EWMH hints | Orchestrator conventions (partially defined) |
| `DISPLAY=:0` / X forwarding | stdin/stdout over SSH |
